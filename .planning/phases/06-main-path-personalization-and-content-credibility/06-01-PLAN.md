# Plan 06-01: Schema Migration — mainLearningPathId and Pending Round State

## Goal
Establish the foundational data model for Phase 6 by adding `mainLearningPathId` FK on `LearnerProfile` and a `PendingOnboardingRound` model for milestone-triggered later rounds. Update `confirmPath` to set the main path when user first selects a recommendation.

## must_haves
- LearnerProfile model has mainLearningPathId nullable FK to LearningPath
- PendingOnboardingRound model exists with userId, roundNumber, dismissCount, createdAt
- confirmPath sets mainLearningPathId on the user's LearnerProfile after creating enrollment
- getMyProfile returns mainLearningPathId in its response
- Prisma migration runs successfully without data loss

## Tasks

<task id="01">
<title>Add mainLearningPathId FK to LearnerProfile and PendingOnboardingRound model in schema.prisma</title>
<read_first>
  - backend/prisma/schema.prisma
</read_first>
<action>
Add the following to the LearnerProfile model in `backend/prisma/schema.prisma`:

```prisma
  mainLearningPathId String? @map("main_learning_path_id")
  mainLearningPath   LearningPath? @relation(fields: [mainLearningPathId], references: [id])
```

Add the reverse relation to the LearningPath model:
```prisma
  mainPathProfiles  LearnerProfile[]
```

Add the following new model after the LearnerProfile model:

```prisma
// ============================================
// PENDING ONBOARDING ROUND (Milestone-triggered)
// ============================================

model PendingOnboardingRound {
  id           String   @id @default(uuid())
  userId       String   @map("user_id")
  roundNumber  Int      @map("round_number")
  dismissCount Int      @default(0) @map("dismiss_count")
  createdAt    DateTime @default(now()) @map("created_at")

  @@unique([userId, roundNumber])
  @@index([userId])
  @@map("pending_onboarding_rounds")
}
```

Note: PendingOnboardingRound intentionally does NOT have a FK relation to User to keep the model lightweight. The userId is used for queries only.
</action>

  - `backend/prisma/schema.prisma` contains `@@map("pending_onboarding_rounds")`
  - `backend/prisma/schema.prisma` contains `dismissCount Int @default(0) @map("dismiss_count")`

<task id="03">
<title>Update confirmPath to set mainLearningPathId on LearnerProfile</title>
<read_first>

  - backend/src/modules/learner-profile/learner-profile.service.ts

<action>
In `backend/src/modules/onboarding/onboarding.service.ts`, update the `confirmPath` method. After the `$transaction` block that creates the `userLearningPath`, add a call to set `mainLearningPathId` on the user's LearnerProfile:

```typescript
// After the userLearningPath creation transaction (line ~401):

// Set this path as main path on LearnerProfile (if profile exists)
// Uses updateMany to silently no-op when profile doesn't exist yet
await this.prisma.learnerProfile.updateMany({
  where: { userId, mainLearningPathId: null },
  data: { mainLearningPathId: dto.learningPathId },
});
```

Use `updateMany` with `mainLearningPathId: null` condition so that:
- First enrollment automatically becomes the main path
- Subsequent enrollments (from Explore) do NOT overwrite the existing main path
- If no LearnerProfile exists (pre-onboarding user), it's a no-op (updateMany returns count: 0)
</action>
<acceptance_criteria>
  - `onboarding.service.ts` contains `mainLearningPathId: null` in a where clause
  - `onboarding.service.ts` contains `mainLearningPathId: dto.learningPathId` in a data clause
  - `onboarding.service.ts` contains `learnerProfile.updateMany` (not update, to avoid throwing on missing profile)
</acceptance_criteria>
</task>
