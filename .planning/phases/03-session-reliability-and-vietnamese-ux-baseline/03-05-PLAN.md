# Plan 05: Fix Explore Enrollment State Inconsistency (Gap Closure)

Explore page uses a page-local `enrolled` state (`useState<Record<string, boolean>>({})`) that resets on navigation. After enrolling on Explore, navigating to Dashboard, and returning to Explore, the local state resets and cards show unenrolled. This plan adds a lightweight backend endpoint that returns the user's enrolled learning path slugs, a frontend hook to query it, and rewires Explore to derive enrollment from server data instead of local state.

**Strategy:** Add `GET /learning-paths/my-enrollments` (protected) that returns enrolled path slugs. Use query key `['learning-paths', 'my-enrollments']` so it auto-invalidates when `useEnroll` invalidates `['learning-paths']` prefix. Explore derives `isEnrolled` from this server-backed Set.

## must_haves

- Backend exposes `GET /api/v1/learning-paths/my-enrollments` as a protected endpoint returning enrolled learning path slugs for the authenticated user
- Frontend `useMyEnrollments` hook queries the new endpoint and returns a `Set<string>` of enrolled slugs
- Explore.tsx does NOT use `useState<Record<string, boolean>>` for enrollment status
- Explore.tsx derives `isEnrolled` from `useMyEnrollments` server data
- Enrollment state persists correctly across Explore -> Dashboard -> Explore navigation without manual refresh
- Query key `['learning-paths', 'my-enrollments']` is automatically invalidated when `useEnroll` fires `invalidateQueries({ queryKey: ['learning-paths'] })`

```typescript
// ── GET /learning-paths/my-enrollments ──────────────────────────────────

/**
 * Trả về danh sách slug của các learning paths mà user đã enroll.
 * Dùng cho Explore page để render enrollment state từ server data.
 *
 * Tại sao trả slug thay vì id?

 * → Trả slug giảm mapping logic phía frontend
 */
async getMyEnrollments(userId: string): Promise<string[]> {
  const enrollments = await this.prisma.userLearningPath.findMany({
    where: { userId },
    select: {
      learningPath: {
        select: { slug: true },
      },
    },
  });

  return enrollments.map((e) => e.learningPath.slug);
}
```

This queries the `user_learning_paths` table joined with `learning_paths` to get slugs. The `select` pattern minimizes data transfer — only the slug string is returned per enrollment.

<action>
Create a new file `frontend/src/hooks/useMyEnrollments.ts`:

```typescript
import { useQuery } from '@tanstack/react-query';
import api from '../services/api';
import { qk } from '../lib/query/queryKeys';
import { useAuthStore } from '../stores/authStore';

/**
 * Fetches the authenticated user's enrolled learning path slugs.
 * Returns a Set<string> for O(1) lookup in Explore cards.
 *
 * Only enabled when user is authenticated (accessToken exists).
 * Auto-invalidated when useEnroll fires because query key
 * ['learning-paths', 'my-enrollments'] is a child of ['learning-paths'].
 */
export function useMyEnrollments() {
  const accessToken = useAuthStore((s) => s.accessToken);

  const query = useQuery({
    queryKey: qk.myEnrollments,
    queryFn: async () => {
      const res = await api.get('/learning-paths/my-enrollments');
      return res.data.data as string[];
    },
    enabled: !!accessToken,
  });

  const enrolledSlugs = new Set(query.data ?? []);

  return { ...query, enrolledSlugs };
}
```

Key design decisions:
- `enabled: !!accessToken` — only fires when user is logged in (Explore is a protected page so this is always true, but guards against edge cases)
- Returns `enrolledSlugs: Set<string>` — O(1) lookup for each card's `isEnrolled` check
- Uses `res.data.data` — matches the standard `{ success, data }` backend response format
</action>
